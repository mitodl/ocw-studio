# Typed settings via django-aqueduct (opt-in)

`ocw-studio` ships two **opt-in** settings modules built on top of
[`django-aqueduct`](https://github.com/mitodl/django-aqueduct) 0.7.0 (codegen
v2), a typed, Pydantic-based settings layer for Django. They exist alongside
the project's normal settings and change nothing unless explicitly selected.

## What's here

- `main/aqueduct_settings.py` — a Pydantic `BaseSettings` model
  (`AqueductSettings`) describing every setting produced by `main/settings.py`
  (and the mitol settings modules it imports), plus a `DevAqueductSettings`
  subclass for local development. The file is generated by codegen v2 and
  carries machine-owned `# >>> aqueduct:generated:*` regions and a hand-written
  `# >>> aqueduct:preserved:validators` region (see below).
- `main/settings_aqueduct.py` — a thin shim that instantiates
  `AqueductSettings` and injects its values into Django via
  `django_aqueduct.configure_django_settings`. Sentry is initialised through
  the `pre_configure` hook so it runs against the validated model before the
  settings are injected.
- `main/settings_aqueduct_dev.py` — the same shim using `DevAqueductSettings`,
  which additionally pulls settings missing from the environment/`.env` file
  from Vault.

## Selecting a settings module

Nothing changes by default. `main/settings.py` remains the settings module
used everywhere (`manage.py`, Celery workers, tests, deployments) unless you
explicitly point `DJANGO_SETTINGS_MODULE` elsewhere:

```shell
# Standard settings (default, untouched):
DJANGO_SETTINGS_MODULE=main.settings

# Aqueduct-based settings, populated from env vars / .env:
DJANGO_SETTINGS_MODULE=main.settings_aqueduct

# Aqueduct-based settings for local dev, additionally reading from Vault
# (configured via VAULT_* env vars) for anything missing from the environment:
DJANGO_SETTINGS_MODULE=main.settings_aqueduct_dev
```

`DevAqueductSettings` builds its Vault source from `VAULT_*` environment
variables via `django_aqueduct.sources.dev.vault_source_from_env`
(`VAULT_ADDR`, `VAULT_PATH`, `VAULT_MOUNT`, `VAULT_KV_VERSION`,
`VAULT_AUTH_METHOD`, `VAULT_TOKEN`, `VAULT_ROLE`). When `VAULT_ADDR` is unset
the source is skipped and it behaves exactly like `AqueductSettings`; when
Vault is enabled but misconfigured it raises a descriptive `VaultConfigError`
rather than a bare `KeyError`. Vault is layered _below_ the environment, so
explicit env vars always win.

## Regenerating the scaffold

Generation is driven by the `[tool.aqueduct]` table in `pyproject.toml`
(modules, output path, parity model/legacy, `parity_ignore`), so the command
is simply:

```shell
python manage.py generate_aqueduct_settings
```

Codegen v2 discovers settings by **static AST analysis** — it never imports
the settings module — and _merges_ into the file's `# >>> aqueduct:generated:*`
regions, leaving the hand-written `# >>> aqueduct:preserved:validators` region
(and anything outside a generated region) untouched. Use `--check` in CI to
fail on drift, and `--reset` to overwrite the whole file from scratch.

Because static discovery cannot see values that only exist at runtime (mitol's
`app_namespaced()` aliases, `init_app_settings()` injections, inline env
reads), the preserved region re-declares those fields with the correct
`validation_alias`, restores the legacy defaults for a few settings, and holds
all the derivations and cross-field validation ported from `main/settings.py`:
`DATABASES`/`CACHES`/Celery broker fallback (via
`django_aqueduct.derivations`), `LOGGING` (driven by `LOG_HOST`/`LOG_HOST_PORT`/
`LOG_LEVEL`/`HOSTNAME`), `INSTALLED_APPS`/`MIDDLEWARE`, `CELERY_BEAT_SCHEDULE`,
`FEATURES` (env `FEATURE_*` merged with source-supplied flags), the
S3-credentials completeness check, the `OCW_EXTRA_THEMES_GTM_IDS` subset check,
the `PLAYWRIGHT_IMAGE_ARCH` enum check, and the idempotent
`GITHUB_APP_PRIVATE_KEY` unicode-escape transform.

`main/aqueduct_settings.py` is excluded from `ruff` (`[tool.ruff]
extend-exclude`) so reformatting never trips `generate_aqueduct_settings
--check`.

## Parity

`python manage.py check_aqueduct_settings` diffs the model's resolved values
against `main/settings.py` and fails on unexplained drift. Deliberate
divergences (and harness artifacts such as the keys Django's
`ConnectionHandler` injects into `DATABASES` after load) are listed with
justifications in `[tool.aqueduct] parity_ignore`.

## Known gaps / follow-ups

- Two fields remain typed `Any` — `NPLUSONE_LOGGER` and `NPLUSONE_LOG_LEVEL`,
  which hold `logging.Logger`/level objects that have no meaningful serialised
  type. Everything else that codegen emitted as `Any` is overridden with a
  concrete type in the preserved region.
